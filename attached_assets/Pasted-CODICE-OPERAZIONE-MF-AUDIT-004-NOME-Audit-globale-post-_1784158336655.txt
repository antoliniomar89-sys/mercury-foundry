OBIETTIVO

Eseguire un audit completo, read-first, dell’attuale Mercury Foundry dopo:

- provider reale OpenAI;
- MF-RUN-003;

- MF-GATE-002;
- isolamento del gate umano;

L’audit deve distinguere rigorosamente:

1. ciò che è realmente implementato;
2. ciò che è soltanto documentato;
3. ciò che è simulato;
4. ciò che manca per un V1 operativo;
5. ciò che impedisce ancora di produrre risultati economicamente utilizzabili.

FASE 0 — VERIFICA GIT E REMOTO

Prima di analizzare il codice:

1. Riporta senza modificare nulla:

- repository remoto configurato;
- URL origin;
- branch locale attivo;
- commit HEAD completo;
- commit origin/main completo;
- stato ahead/behind;
- working tree pulito o sporco;
- elenco degli ultimi 15 commit;
- directory radice effettivamente versionata;
- eventuali repository Git annidati;
- eventuali file esclusi da .gitignore;
- presenza su Git dell’ultima implementazione di:
  - human_gate.py;
  - test_human_approval_protocol.py;
  - test_gate_isolation.py;
  - gestione approval_revoked;
  - incident report;
  - provider reale;
  - doctor aggiornato.

2. Verifica esplicitamente perché il repository pubblico può ancora mostrare:

- README “Mercury Foundry V0”;
- FakeModel come unico provider;
- vecchio comando approve;
- cronologia apparentemente limitata.

3. Se HEAD locale e origin/main non coincidono, fermati e riporta il problema.
Non effettuare automaticamente push, force-push, rebase o merge.

FASE 2 — AUDIT DI SICUREZZA E INTEGRITÀ

6. Verifica almeno:

- nessun fallback silenzioso a FakeModel;
- identità provider e is_simulated persistiti;
- budget chiamate/token/costo fail-closed;
- path traversal e symlink escape;
- isolamento staging/target;
- atomicità della promozione;
- rollback compensativo;
- append-only reale dell’audit;
- consistenza transazionale DB/filesystem;
- recovery dopo crash;
- idempotenza approve/reject/revoke;
- concorrenza fra candidate;
- impossibilità per submit/verify/evaluator/diagnostics/test di approvare;
- approvazione disabilitata di default;
- assenza di secret nei log, DB, audit e report;
- challenge monouso e scadenza;
- impossibilità di riutilizzo della challenge;
- comportamento in TTY, non-TTY, pytest, CI e Agent;
- target_project vuoto;
- candidate 2 ancora approval_revoked;
- storia dell’incidente intatta.

7. Cerca esplicitamente:

- funzioni non più usate;
- vecchi percorsi di approvazione ancora importabili;
- bypass indiretti;
- configurazioni duplicate;
- migrazioni fragili;
- documentazione falsa o obsoleta;
- file “Pasted-*”, artefatti Agent o memory accidentalmente versionati;
- dati runtime/DB che non dovrebbero essere pubblicati;
- chiavi, token o frammenti di secret nella cronologia Git.

Non stampare eventuali secret trovati: riporta solo file, tipo di rischio e azione necessaria.

14. Valuta separatamente:

capacità di trasformare una specifica in candidate testata.

B. Mercury economico:
capacità di osservare mercato, scegliere opportunità, validare domanda, vendere, consegnare e misurare outcome economici.

15. Non confondere le due cose.
